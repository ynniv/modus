# i386: 32-bit x86 on Bare Metal

## Overview

Modus runs on i386 (32-bit x86) as a Multiboot kernel under QEMU. The i386 port is the most constrained target: 8 general-purpose registers, 30-bit fixnums, and no dedicated allocation registers. Despite this, i386 achieves full self-hosting, SSH networking via an NE2000 ISA NIC, and participates as a third architecture in the cross-compilation fixpoint proof.

```bash
# Build and run REPL
sbcl --script mvm/build-i386-repl.lisp
qemu-system-i386 -kernel /tmp/modus64-i386.bin -m 256 \
  -display none -serial stdio -no-reboot

# Build and run SSH
sbcl --script mvm/build-i386-ssh.lisp
qemu-system-i386 -m 256 -nographic -no-reboot \
  -kernel /tmp/modus64-i386-ssh.bin \
  -device ne2k_isa,netdev=net0,iobase=0x300,irq=9 \
  -netdev 'user,id=net0,hostfwd=tcp::2222-:22'

# Connect
ssh -p 2222 -o StrictHostKeyChecking=no test@localhost
```

## How i386 Fits into the Architecture

Modus targets 9 CPU architectures through the MVM (Modus Virtual Machine) — a portable bytecode ISA with per-architecture native code translators. The i386 translator (`mvm/translate-i386.lisp`) converts MVM bytecodes to 32-bit x86 native code.

The architecture matrix looks like this:

```
                  ┌─────────┐
                  │  SBCL   │  (Gen0 seed compiler, x64 only)
                  └────┬────┘
                       │
              ┌────────┼────────┐
              ▼        ▼        ▼
           ┌─────┐  ┌─────┐  ┌─────┐
           │ x64 │  │AArch│  │i386 │  ← MVM translators
           └──┬──┘  └──┬──┘  └──┬──┘
              │        │        │
              └────────┼────────┘
                       │
              Any → Any cross-compilation
```

Each architecture can cross-compile to any other. The fixpoint proof (`scripts/run-fixpoint-i386.sh`) runs 8 QEMU steps to verify that x64, AArch64, and i386 all produce byte-identical output for the same bytecodes, regardless of which architecture runs the translator.

### Shared code, per-architecture adapters

The networking stack (`net/`) is architecture-independent. Each platform provides a thin adapter defining I/O access, memory addresses, and NIC driver bindings:

| Layer | x64 | AArch64 | i386 |
|-------|-----|---------|------|
| NIC | E1000 (PCI MMIO) | E1000 (PCI ECAM) | NE2000 (ISA port I/O) |
| PCI | I/O ports 0xCF8/0xCFC | ECAM at 0x3F000000 | None (ISA bus) |
| Adapter | `net/arch-x86.lisp` | `net/arch-aarch64.lisp` | `net/arch-i386.lisp` |
| Crypto | `net/crypto.lisp` | `net/crypto.lisp` | `net/crypto-32.lisp` + `net/crypto-i386.lisp` |

The NIC driver presents the same `e1000-send`/`e1000-receive`/`e1000-rx-buf` API regardless of underlying hardware. On i386, `ne2000.lisp` implements this API via NE2000 register I/O, and the shared `ip.lisp`/`ssh.lisp` code runs unmodified.

## Boot Sequence

i386 boots via Multiboot1 — the same protocol used by GRUB. QEMU's `-kernel` flag loads the ELF directly.

```
boot/boot-i386.lisp
  1. Multiboot1 header at 0x100000 (1MB)
  2. QEMU hands off in 32-bit protected mode (A20 enabled, paging off)
  3. Set up stack at 0x400000 (4MB)
  4. Initialize alloc state at fixed addresses:
       0x600: alloc pointer  → 0x00800000 (8MB, cons space)
       0x604: alloc limit    → 0x04000000 (64MB, general heap)
       0x608: NIL constant   → 0x00000000
  5. Initialize COM1 serial (115200 8N1)
  6. Fall through to compiled native code
```

This is simpler than x64 (no long mode transition, no page tables) — 32-bit flat protected mode is sufficient for identity-mapped operation with 256MB of RAM.

### NIL = 0

On i386, NIL is fixnum 0. This means `(zerop x)` and `(null x)` are equivalent — a value of 0 is both numerically zero and logically false. This matches the AArch64 and RISC-V convention but differs from x64 where NIL is `#xDEAD0001`.

## Memory Layout

```
0x00100000  Kernel image (native code, Multiboot load address)
0x00200000  Network state (e1000-state-base: MAC, IP, ARP, crypto K tables)
0x00280000  SSH connection buffers (ssh-conn-base)
0x00300000  SSH IPC shared state (ssh-ipc-base: line editor, eval, capture)
0x00360000  Per-CPU data (32 bytes/CPU)
0x00400000  Stack (grows down)
0x00780000  NE2000 RX/TX host buffers
0x00800000  Cons space (bump-allocated)
0x04000000  General heap (64MB)
```

All state addresses are below 8MB (the cons base), fitting comfortably in 30-bit fixnum range.

## 30-bit Fixnum Constraints

The defining challenge of i386. With fixnum-shift=1 and 32-bit words, positive fixnums max out at 2^30 - 1 ≈ 1.07 billion. Any value above this overflows when tagged — `(ash byte 24)` overflows for byte >= 64 because the tagged representation `(byte << 24) << 1` exceeds 32 bits.

This affects three subsystems:

### Networking (u32 values)

TCP sequence numbers, IP addresses, and SSH packet lengths are big-endian u32 values. Reading `(ash byte 24)` for the high byte overflows.

**Fix**: `net/i386-overrides.lisp` masks the high byte to 6 bits:
```lisp
(defun buf-read-u32-mem (addr off)
  (logior (ash (logand (mem-ref (+ addr off) :u8) #x3F) 24)   ; 6-bit high byte
          (logior (ash (mem-ref (+ addr off 1) :u8) 16)
                  (logior (ash (mem-ref (+ addr off 2) :u8) 8)
                          (mem-ref (+ addr off 3) :u8)))))
```

TCP sequence numbers lose the top 2 bits but relative offsets still work correctly.

### Cryptography (32/64-bit arithmetic)

SHA-256 constants like `0xb5c0fbcf`, ChaCha20 quarter-round additions, and Poly1305 multiply-accumulate all exceed 30 bits. Two layers of pair arithmetic handle this:

1. **`net/crypto-32.lisp`** — Field multiplication for X25519/Ed25519. Uses `(hi26 . lo26)` pairs: 26-bit halves representing 52-bit products. The ×19 reduction factor uses shift-and-add (19 = 16 + 2 + 1) to avoid overflow.

2. **`net/crypto-i386.lisp`** — SHA-256, SHA-512, ChaCha20. Uses `(hi16 . lo16)` pairs: 16-bit halves representing full 32-bit values. Each half fits safely in a fixnum (max 65535 << 1 = 131070). Reimplements all rotate/shift/add operations at the pair level.

### Fixpoint metadata (image assembly)

The fixpoint translator writes u32 metadata (function table entries, MVMT magic `0x544D564D`, kernel-main hash). Values with byte 3 >= 0x80 overflow when assembled via `td-read-u32` + `img-emit-u32`.

**Fixes**:
- Raw byte copy for function table name hashes (bypass arithmetic)
- Raw 4-byte copy for kernel-main hash
- MVMT magic written as 4 individual byte literals

## NE2000 ISA NIC

i386 uses an NE2000 ISA NIC instead of E1000 PCI. The NE2000 is a simple 8-bit ISA device accessed entirely through I/O ports — no PCI bus, no MMIO, no DMA engine. All register values are below 0xFFFF, safely within fixnum range.

```
net/ne2000.lisp
  QEMU: -device ne2k_isa,netdev=net0,iobase=0x300,irq=9
  I/O base: 0x300 (registers 0x00-0x0F, data port 0x310, reset 0x31F)
  NIC RAM: pages 0x40-0xBF (32KB), 256 bytes/page
  TX buffer: page 0x40 (1536 bytes)
  RX ring: pages 0x46-0xBF
```

The MVM compiler requires I/O port numbers to be compile-time constants. Since `io-in-byte`/`io-out-byte` can't take computed port arguments, `ne2k-read`/`ne2k-write` use 16-way `if` chains dispatching on register index:

```lisp
(defun ne2k-read (reg)
  (if (eq reg 0) (io-in-byte #x300)
  (if (eq reg 1) (io-in-byte #x301)
  ...
  (io-in-byte #x30F)))))))))))))))))
```

Packet send/receive uses remote DMA: bytes are transferred one at a time through the data port at 0x310. The ring buffer wraps at `rx-stop` (page 0xC0), requiring two-part reads for packets spanning the boundary.

## i386 Translator

`mvm/translate-i386.lisp` translates MVM bytecodes to 32-bit x86 native code. Key constraints:

### Register allocation

Only 6 usable GPRs (ESP/EBP reserved for stack):

| Virtual reg | Physical | Role |
|-------------|----------|------|
| V0 | ESI | General (callee-saved) |
| V1 | EDI | General (callee-saved) |
| V4 | EBX | General (callee-saved) |
| VR | EAX | Return value / accumulator |
| — | ECX | Scratch (shift counts) |
| — | EDX | Scratch (multiply, division) |

Allocation state (alloc pointer, limit, NIL) lives in memory at fixed addresses (0x600-0x608) rather than in dedicated registers. Every allocation requires load/store through memory.

### Spill slots

Functions with more than 3 temp variables (V0, V1, V4) spill to the stack frame. The translator supports 7 spill slots (V9-V15). Functions exceeding this must be split into smaller helpers — a constraint that shaped the fixpoint translator's structure.

### Calling convention

4-byte stack slots. Arguments beyond 4 are passed on the stack. Callee-saved registers (ESI, EDI, EBX) are preserved across calls. The frame pointer (EBP) anchors both local variables and spill slots.

## Cross-Compilation Fixpoint

i386 participates in all 5 fixpoint proofs:

```
1. SHA256(Gen1) == SHA256(Gen3)        x64↔AArch64 fixpoint
2. SHA256(i386-A) == SHA256(i386-B)    i386 translator determinism
3. SHA256(i386-A) == SHA256(i386-C)    i386 self-hosting fixpoint
4. SHA256(x64-D) == SHA256(Gen2)       i386→x64 cross-compilation
5. SHA256(AArch64-E) == SHA256(Gen1)   i386→AArch64 cross-compilation
```

Proofs 2-5 involve i386 directly. The self-hosting proof (#3) means i386 running the MVM translator produces the same i386 binary that x64 produced — the translator is a fixed point of itself. Proofs 4-5 mean i386 can cross-compile to x64 and AArch64, producing byte-identical images to those architectures' own output.

### i386→x64 cross-compilation

Three categories of fixes were needed:

1. **Separate translation pipeline** — Overriding `td-translate-fn-body` via last-defun-wins crashed x64→x64 translation. Fix: separate pipeline with distinct function names (`translate-mvm-to-x64-from-i386`, `td-translate-fn-body-li-safe`).

2. **LI opcode raw byte emission** — `decode-u64` overflows on i386 (ash by 32 is a no-op on 32-bit x86). The safe pipeline intercepts LI (opcode 17) before `decode-instruction` and emits x64 `MOV reg, imm64` by copying 8 raw bytes from bytecodes.

3. **boot-x64.lisp literal overflow** — Constants like `#x80000000` (CR0.PG) and `#xDEAD0001` (NIL) exceed 30-bit range. Changed to emit as individual bytes (safe for all architectures).

### i386→AArch64 cross-compilation

Three categories of fixes:

1. **Vreg spill mismatch** — On bare-metal x64/AArch64, `make-array` ignores `:initial-element`, so vreg-to-phys slots contain fixnum 0 (truthy), not nil. The i386 translator must match this behavior: never spill V9-V15.

2. **`&key` parameter mismatch** — MVM converts `&key (hw 0)` to a positional param with a nil-check default. Uninitialized params are garbage on x64/AArch64 (non-nil, default skipped) but 0=nil on i386 (default triggers). Fix: changed all `&key` params in `translate-aarch64.lisp` to positional.

3. **Multi-arg logior** — `(logior a b c d)` broken on bare metal (same class as "3-arg + is broken"). The ADR fixup resolution used 4-arg logior. Fix: nested 2-arg calls.

## Build Scripts

| Script | Output | Description |
|--------|--------|-------------|
| `mvm/build-i386-repl.lisp` | `/tmp/modus64-i386.bin` | Serial REPL |
| `mvm/build-i386-ssh.lisp` | `/tmp/modus64-i386-ssh.bin` | SSH server (NE2000) |
| `mvm/build-fixpoint.lisp` | `/tmp/fixpoint-gen0.elf` | Fixpoint seed (includes i386 translator) |

Source load order for SSH:
```
arch-i386.lisp → ne2000.lisp → ip.lisp → crypto.lisp → crypto-32.lisp →
crypto-i386.lisp → ssh.lisp → aarch64-overrides.lisp → i386-overrides.lisp
```

Later files override earlier defuns via last-defun-wins. `i386-overrides.lisp` must load last to replace 64-bit-unsafe functions with 30-bit-safe versions.
